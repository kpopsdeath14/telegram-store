# Автоматическое обновление трек-номеров СДЭК

## Что было реализовано

Простая и эффективная система автоматического получения трек-номеров для конкретного заказа СДЭК.

### Изменения в шаблонах (templates)

#### Файл: `services/cdek.clj`

Добавлены функции для автоматического получения трек-номера **для конкретного заказа**:

- `extract-track-number-from-response` - извлечение трек-номера из ответа СДЭК API
- `set-order-track-number!` - сохранение трек-номера в БД
- `try-fetch-track-number` - одна попытка получить трек-номер из СДЭК
- `schedule-track-number-retries` - планирование отложенных попыток для конкретного заказа

#### Файл: `core.clj`

Обновлена функция `fetch-track-number!`:
- Если трек-номер получен сразу → сохраняется в БД
- Если трек-номер НЕ получен сразу → запускаются отложенные попытки **только для этого заказа**

---

## Как это работает

### Сценарий 1: Трек-номер доступен сразу ✅

1. Создается заказ в СДЭК
2. Сразу запрашивается информация о заказе
3. Трек-номер найден → сохраняется в БД
4. **Готово!**

### Сценарий 2: Трек-номер появляется позже ⏱️

1. Создается заказ в СДЭК
2. Сразу запрашивается информация о заказе
3. Трек-номер НЕ найден → запускаются отложенные попытки:
   - **Через 5 минут** - попытка №1
   - **Через 10 минут** - попытка №2
   - **Через 20 минут** - попытка №3
   - **Через 30 минут** - попытка №4
   - **Через 60 минут** - попытка №5
4. Как только трек-номер найден → сохраняется в БД и попытки прекращаются
5. **Готово!**

---

## Преимущества решения

✅ **Простота** - минимум кода, легко понять
✅ **Эффективность** - проверяется только конкретный заказ, а не все заказы
✅ **Автономность** - каждый заказ управляет своими попытками
✅ **Автоматическая остановка** - как только трек-номер получен, попытки прекращаются
✅ **Не требует БД** - не нужны дополнительные SQL функции
✅ **Не требует глобальных планировщиков** - каждый заказ имеет свой планировщик

В файле `services/cdek.clj` найдите функцию `schedule-track-number-retries`:

```clojure
(defn schedule-track-number-retries [order-id cdek-uuid payment-id]
  (let [delays [5 10 20 30 60]  ; минуты - ИЗМЕНИТЕ ЗДЕСЬ
        executor (java.util.concurrent.Executors/newSingleThreadScheduledExecutor)]
    ...))
```

Примеры настройки:

```clojure
;; Более частые проверки (для тестирования)

;; Более редкие проверки (для экономии API запросов)
```

## Тестирование

### 1. Ручное тестирование

1. Создайте заказ с доставкой СДЭК через приложение
2. Проверьте логи - должны появиться сообщения о запланированных попытках
3. Подождите несколько минут
4. Проверьте логи - должны появиться попытки получения трек-номера
5. Проверьте БД - трек-номер должен появиться в заказе

### 2. Проверка в тестовом режиме СДЭК

В тестовом API СДЭК трек-номера могут появляться быстрее или медленнее, чем в продакшн.

### 3. Симуляция задержки (для разработки)

## Архитектура

### Независимые планировщики для каждого заказа

Каждый заказ создает **свой собственный** `ScheduledExecutorService`:
- Планировщик создается только для заказов без трек-номера
- Планировщик автоматически останавливается после успешного получения трек-номера
- Планировщик автоматически завершается после всех попыток

### Хранение данных

**CDEK UUID** сохраняется в `exchange_history`:
```json
{
  "order_create": {...},
  "order_info": {...},
  "order_uuid": "abc-123",
  "track_number": null
}
```

**Трек-номер** сохраняется в таблице заказов:
```sql
CALL "order".order_track_number_set(_p := {
  order_id: 123,
  track_number: "1234567890"
});
```

## Производительность

- **Потребление памяти:** минимальное - один легковесный планировщик на заказ без трек-номера
- **Нагрузка на API СДЭК:** 1-5 запросов на заказ (в зависимости от когда появится трек-номер)
- **Задержка:** от 0 секунд (если трек-номер доступен сразу) до 60 минут (последняя попытка)
- **Автоматическая очистка:** планировщики автоматически останавливаются и освобождают ресурсы

## Troubleshooting

### Трек-номера не обновляются

**Проблема:** Логи показывают попытки, но трек-номер не появляется

**Решения:**
1. Проверьте, что СДЭК API доступен
2. Убедитесь, что CDEK credentials корректны
3. Проверьте статус заказа в личном кабинете СДЭК
4. Увеличьте количество попыток или интервалы

### Попытки не запускаются

**Проблема:** Не видно логов с `[CDEK][retry]`

**Решения:**
1. Убедитесь, что трек-номер действительно не получен при создании заказа
2. Проверьте логи на наличие сообщения "scheduling retries"
3. Проверьте, что UUID заказа СДЭК корректно извлечен

### Слишком много попыток

**Проблема:** Хотите уменьшить нагрузку на API

**Решение:** Уменьшите количество попыток или увеличьте интервалы:
```clojure
[10 30 60]  ; только 3 попытки
```
